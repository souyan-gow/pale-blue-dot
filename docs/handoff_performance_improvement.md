## 1. プロジェクト概要

「Pale Blue Dot Delivery」は、カール・セーガンの「ペイル・ブルー・ドット」にインスパイアされたアンビエント宇宙探索ゲームのプロトタイプである。単一HTMLファイル（Three.js r128）で動作し、太陽系の惑星を自由に探索・着陸・写真撮影できる。

**開発工程の成果**:
- 全惑星（水星〜海王星）＋月＋αケンタウリ系の実装完了
- 地表ビュー（6層パララックス）全惑星対応
- 写真撮影・アルバム・IndexedDB永続化
- Web Audio APIによるアンビエントサウンド
- v2.0テスト仕様書に基づく4回のテストサイクル完了（R-01〜R-58）

**主要ファイル**:
- `src/pale_blue_dot_prototype.html` — ソースコード（約2,920行）
- `index.html` — 本番用コピー（src と常に同期）
- `docs/PALE_BLUE_DOT_修正要望管理表.xlsx` — 全修正履歴

## 2. 性能向上への申し送り事項

### 2.1 【必須】惑星ズーム距離の均一化

**現状の問題**:
惑星をクリックしてズームする際の距離が不均一である。木星・土星など大きな惑星はある程度近くまでズームできるが、水星・地球などの小さな惑星は遠い地点でズームが止まってしまう。

**根本原因**:
`navigateTo()` 関数（約1,463行目）のズーム距離計算式が線形であるため。

```javascript
const targetR = pd.r > 0 ? pd.r * 12 + 30 : pd.orbitR * 0.5;
```

この式では、惑星半径 `pd.r` に対して線形に距離が増えるため、大きな惑星ほど相対的に近くに見え、小さな惑星はスクリーン上で非常に小さく表示される。

**各惑星のズーム距離（現状）**:

| 惑星 | 半径(r) | ズーム距離 | 画面上の見え方 |
|------|---------|-----------|--------------|
| 水星 | 0.383 | 34.6 | 非常に小さい |
| 金星 | 0.950 | 41.4 | 小さい |
| 地球 | 1.000 | 42.0 | 小さい |
| 火星 | 0.532 | 36.4 | 非常に小さい |
| 木星 | 10.973 | 161.7 | 適切 |
| 土星 | 9.140 | 139.7 | 適切 |
| 天王星 | 3.981 | 77.8 | やや小さい |
| 海王星 | 3.865 | 76.4 | やや小さい |

**推奨修正方針**:
視野角ベースの計算に変更し、すべての惑星が画面上でほぼ同じ大きさに見えるようにする。

```javascript
// 例: 惑星が画面の一定割合を占めるようにズーム距離を計算
const desiredAngularSize = 0.15; // ラジアン（約8.6度）
const targetR = pd.r / Math.tan(desiredAngularSize / 2);
```

あるいは、対数スケールで補正する方法も有効：

```javascript
const targetR = pd.r * (8 + 40 / Math.max(pd.r, 0.5)) + 10;
```

**影響範囲**: `navigateTo()` 関数のみ。カメラ制御の他の部分（`updateCamera()`、`spherical` 変数）には影響しない。

### 2.2 【必須】全惑星の地表描画の精巧化

**現状の問題**:
地表に降り立った際の描画（`drawSurfaceView()` 関数、約2,447〜2,840行目）が、全惑星でシンプルなパララックスレイヤー構成となっており、リアリティに欠ける。

**現在の地表描画の仕組み**:
- Canvas 2D APIで6層のパララックスレイヤーを描画
- 各惑星固有のカラーパレットと地形パラメータを `configs` オブジェクトで定義
- fbmノイズで地形シルエットを生成
- マウスドラッグで左右にパン（コックピット視点）

**改善提案**:

| 項目 | 現状 | 改善案 | 優先度 |
|------|------|--------|--------|
| 大気表現 | なし | 遠方レイヤーに霞み（大気散乱）を追加。距離に応じて彩度低下・明度上昇 | 高 |
| 地形ディテール | fbmノイズのみ | クレーター・峡谷・岩石などのディテールテクスチャを重畳 | 高 |
| 惑星固有の演出 | 土星の嵐のみ | 火星の砂嵐、金星の硫酸雲、木星の大赤斑渦などを追加 | 中 |
| 空の表現 | 単色グラデーション | 時間変化する空（薄明、星の瞬き）を追加 | 中 |
| モーションブラー | なし | パン時の微弱なブラーで臨場感向上 | 低 |
| 地面テクスチャ | シルエットのみ | 前景レイヤーに岩肌・砂地などのテクスチャを追加 | 中 |

**テクスチャ参照元**: `src/solar_system_visualizer.html`（1,230行）に高品質テクスチャの実装例あり。

**影響範囲**: `drawSurfaceView()` 関数内の `configs` オブジェクトと描画ロジック。3Dシーンには影響しない。

### 2.3 テクスチャ品質の向上（3D惑星）

**現状**:
11種類のプロシージャルテクスチャが512x512解像度で生成されている（`texMercury()`〜`texNeptune()`、362〜706行目）。4Kモニターではピクセルが目立つ。

**改善提案**:
- 解像度を512x512 → 1024x1024に引き上げ（メモリ使用量4倍、起動時間増加に注意）
- fbmノイズのオクターブ数を増やし、細部のディテールを追加
- 水星: クレーターのハイライトとシャドウを強調
- 地球: 雲テクスチャの改善（現在の `texEarthCloud()` は均一すぎる）
- 土星の輪: 微細な縞模様とカッシーニの間隙を追加
- 星空: 2048x1024 → 4096x2048に引き上げ、星の輝度バリエーションを増やす

**注意**: 解像度増加は起動時間に直結する。`loadAllTextures()` のプログレスバー表示は実装済み（711〜735行目）だが、体感待ち時間の増加を考慮すること。

---

### 2.4 起動時パフォーマンスの最適化

**現状の問題**:
11種のテクスチャを同期的に順次生成しているため、起動に数秒かかる。

**改善提案**:
- **ノイズ関数の最適化**: `smoothNoise()` / `fbm()`（346〜353行目）が数百万回呼ばれる。ルックアップテーブル方式に変更することで高速化可能
- **テクスチャのプリベイク**: 初回起動時に生成したテクスチャをIndexedDBにキャッシュし、2回目以降は読み込みのみ
- **Web Worker**: テクスチャ生成をWeb Workerに移し、メインスレッドのブロッキングを解消
- **ジオメトリのLOD**: 小さな惑星のスフィアセグメント数を削減（現在は全惑星48セグメント固定）

---

### 2.5 レンダリングループの最適化

**現状**:
`animate()` 関数（約2,835〜2,891行目）は毎フレーム全惑星の位置・回転を更新している。

**改善提案**:
- 視錐台カリングの活用: 画面外の惑星は位置更新をスキップ
- 距離ベースの更新頻度制御: 遠い惑星は2〜3フレームに1回の更新に削減
- 地球の雲回転: 毎フレームではなく3フレームに1回に変更

---

### 2.6 オーディオシステムの改善

**現状の問題**:
Web Audio APIのオシレーター合成で宇宙アンビエントを生成しているが（747〜812行目）、音量が小さい（R-30, R-31で指摘済み）。

**改善提案**:
- マスターゲイン値の再調整（現在の `masterGain` / `ambientGain` / `proxGain` の見直し）
- マスターリミッターの追加（複数音源の同時再生時のクリッピング防止）
- プリレコーデッド音源への移行検討（合成音よりファイルサイズは増えるが品質向上）

---

### 2.7 コードアーキテクチャの改善（中長期）

**現状**:
約2,920行のJavaScriptが単一の `buildScene()` 関数内に格納されたモノリシック構造。

**改善提案**（優先度の高い順）:
1. **設定の外部化**: 惑星データ（`PLANETS` 配列）、地表設定（`configs`）をJSONまたは別JSファイルに分離
2. **モジュール分割**: テクスチャ生成、オーディオ、写真システム、地表ビューをそれぞれ別ファイルに分離
3. **状態管理**: `camState` を中心としたゲーム状態の一元管理化
4. **テストの導入**: ズーム計算、写真保存/読込、状態遷移のユニットテスト

**注意**: 現在の単一ファイル構造は「インストール不要でindex.htmlだけで動作する」というプロジェクト要件に基づいている。モジュール分割にはバンドラー（Vite等）の導入が前提となる。

## 5. 開発工程で得られた知見

1. **drawIconFrame() の描画座標**: Canvas 2D の `drawImage()` 第5〜8引数（描画先座標）はクリップ領域との整合性に注意。R-40 → R-52 → R-58 と3回の修正を要した。
2. **月の軌道実装**: 地球の `rotation.z`（自転軸傾き）が子オブジェクトに継承されるため、月の軌道リングは `moonOrbitPivot`（独立Object3D）で実装。毎フレーム地球の位置をコピーしている。
3. **潮汐固定の実装**: `m.rotation.y = Math.PI - a` で軌道角度に応じた回転を直接設定。`rotSpeed: 0.0` としてanimate内の汎用回転処理を無効化。
4. **テスト→修正サイクル**: 4回のテストサイクルでR-01〜R-58の58件を処理。Excel管理表とgitの連携運用が有効だった。
5. **PostToolUseフック**: `src/pale_blue_dot_prototype.html` の編集時に自動的に `index.html` へコピーするフックが有効に機能した（`.claude/hooks/sync-index.py`）。
